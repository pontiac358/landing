**Diálogo** se utiliza para mostrar contenido secundario y para centrar la atención del usuario en la realización de una acción o escenario específico. El diálogo bloquea la interacción del usuario con la página principal hasta que se cierre o se complete la acción deseada dentro de él.

### Punto de entrada

El punto de entrada a un diálogo puede ser un clic en un disparador específico (por ejemplo, un botón en la interfaz) o acciones o condiciones particulares que surjan en la interfaz (por ejemplo, intentar salir de la página sin guardar datos, o cambiar de cuenta en otra pestaña).

### Tamaños

El tamaño del diálogo se selecciona en función del contenido.

![Sizes](/static/images/design/Dialog/sizes.png)

**Tamaño S (480 px)** se utiliza para advertencias o confirmaciones pequeñas, así como para formularios pequeños.

**Tamaño M (720 px)** se utiliza para formularios pequeños y medianos.

**Tamaño L (900 px)** se utiliza para datos masivos y flujos dentro de una ventana modal.

**Tamaño personalizado** se utiliza cuando ninguna de las tres opciones (`S`, `M` o `L`) es adecuada.

### Área de contenido

El contenido dentro del diálogo puede ser arbitrario. El diseño se selecciona según las necesidades.

Si el diálogo contiene un formulario, se recomienda utilizar uno de los dos tipos de disposición de elementos:

### Posicionamiento del diálogo en la página

El diálogo se alinea al centro de la pantalla tanto vertical como horizontalmente.

Al escalar el viewport, el posicionamiento se mantiene y el tamaño del diálogo se reduce, siempre que no interfiera con el contenido interno.

El margen mínimo permitido desde el borde superior e izquierdo de la pantalla es de 20 px.

![position](/static/images/design/Dialog/position.png)

Si la ventana no cabe en el viewport, los márgenes superior e izquierdo permanecerán fijos, y el diálogo se extenderá más allá de los bordes derecho e inferior de la pantalla. Este es un estado no deseado, y se recomienda diseñar el diálogo de manera que quepa en el viewport en cualquier pantalla, con márgenes de al menos 20 px desde el borde de la pantalla en todos los lados.

If the dialog is non-blocking, it can be closed in the following ways:

- By clicking outside the modal window.
- By pressing the Esc key.

The "Cancel" button and the close icon are familiar elements, as they are universally used in dialogs for closing. Clicking outside the modal window helps the user close it more easily and quickly, as they don't need to aim for the first two elements. Pressing the Esc key allows closing the window without using the mouse.

Since interaction is only intended with the dialog window, the content of the parent page should not scroll.

### Protection Against User Data Loss

Due to the ease of closing a modal window, there's a possibility of doing so accidentally. If the user was only selecting checkboxes in the modal window, losing such data is easier to recover from than when the user has been typing code for a long time or re-entering a 12-digit car VIN. We must prevent such situations. At the same time, we should not prevent the user from closing the modal window in their usual way.

To avoid data loss errors, it's best to use an **additional data deletion confirmation window**. In this case, the user will need to confirm the action.

![Pop-up S.png](/static/images/design/Dialog/Pop-up_S%202.png)

Whether to add another confirmation window when closing a modal window should be assessed based on common sense. It's important to strike a balance between security and potential annoyance from too frequent prompts. The goal of confirmation is not to lose the data entered by the user, without hindering them from completing their task.
