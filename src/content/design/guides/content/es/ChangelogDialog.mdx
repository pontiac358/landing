Con el changelog, el equipo del servicio puede informar a los usuarios sobre los cambios realizados.

El changelog se muestra una sola vez al abrir el servicio. No hay una opción para volver a abrir el changelog; si es necesario, puedes implementarla tú mismo.

Dado que el changelog se muestra en una ventana modal, se aplican las reglas de cierre y escalado para ventanas modales.

## Elementos

El changelog es una ventana modal de tamaño `M` con publicaciones ordenadas cronológicamente. Cada publicación tiene:

1. fecha de publicación;
2. [etiqueta](https://gravity-ui.com/design/guides/label) `New` (opcional);
3. título;
4. imagen (opcional);
5. texto;
6. [botón](https://gravity-ui.com/design/guides/button) de acción (opcional).

![Elements](/static/images/design/ChangelogDialog/ChangelogDialog1.png)

### Fecha de publicación

La fecha de publicación permite al usuario entender cuándo se realizó un cambio específico en el servicio. Por defecto, la fecha se muestra en formato `DD Mon YYYY`, pero se pueden usar otros formatos de fecha teniendo en cuenta la localización del servicio.

### Etiqueta

La etiqueta `New` se utiliza para marcar las publicaciones nuevas. Qué publicaciones se consideran nuevas y se marcan con esta etiqueta lo determina el servicio. Es posible usar otras formulaciones de texto distintas de `New`.

### Título

El título de la publicación se utiliza para describir brevemente un cambio en el servicio. El título puede tener cualquier longitud. No se recomienda:

- usar títulos que ocupen más de una línea (que contengan más de 70 caracteres), ya que esto empeora la legibilidad del texto;
- duplicar el texto de la noticia en el título.

Para el título se utiliza el estilo `text-subheader-3`. No se prevé el uso de otros estilos de texto.

### Imagen

Aquí tienes la traducción del archivo fuente de la guía de diseño al español, buscando que suene lo más natural posible:

La imagen es opcional y tiene unas dimensiones fijas de 516x240 píxeles. Se pueden utilizar archivos con extensión JPG y PNG como imagen; no se admiten imágenes GIF ni archivos de vídeo. La imagen se utiliza cuando es necesario ilustrar el tema de una publicación. Se utiliza una única imagen tanto para el tema claro como para el tema oscuro de la interfaz.

Si se carga una imagen que supere las dimensiones de 516x240 píxeles, se escalará y se alineará al centro del lado más largo.

Teniendo en cuenta las pantallas Retina, se recomienda cargar imágenes con un tamaño duplicado en cada lado, es decir, 1032x480 píxeles.

### Texto

Después de la imagen, se encuentra el texto de la publicación, que puede tener una longitud arbitraria. No se recomienda:

- Utilizar textos muy largos (más de 500 caracteres). Si no es posible acortar el texto a la longitud recomendada, se pueden añadir enlaces a la documentación o hacer que el texto completo de la publicación sea accesible haciendo clic en un botón después del texto.
- Crear subtítulos. Dado que la longitud recomendada del texto no implica que vaya a ser excesivamente largo, no se requiere una división adicional en bloques mediante subtítulos.

Para el texto se utiliza el estilo `text-body-1`. No se contempla el uso de otros estilos de texto, pero sí es posible el formato: creación de listas con viñetas y numeradas, tachado y subrayado de texto, así como el uso de negrita y cursiva.

### Botón de acción

El botón de acción es opcional y se sitúa debajo del texto de la publicación. Se utiliza para acceder al texto completo de la publicación.

Solo se permite el uso de botones de tamaño `M`. El botón puede ser de cualquier tipo, con o sin icono. Se recomienda utilizar un único tipo de botón.

No se contempla la presencia de varios botones.

### Otros elementos

Debajo del título de la ventana modal, también puede haber un enlace `View full changelog` que dirija a la página del changelog; esta puede ser una página separada del servicio, un Storybook, un servicio de terceros con documentación, etc.

## Recomendaciones de uso

El registro de cambios se aplica:

- Cuando es necesario informar al usuario sobre cambios significativos que han ocurrido en el servicio.
- Cuando es necesario mantener al usuario informado sobre cambios menores. En este caso, se debe mostrar el registro de cambios periódicamente e informar sobre todos los pequeños cambios desde la última entrada.
- Cuando se desea recordar al usuario sobre los cambios en el servicio y aumentar su conocimiento sobre lanzamientos anteriores.

No se recomienda:

- Usar el registro de cambios para la incorporación de usuarios (onboarding).
- Usar el registro de cambios para informar al usuario sobre cada pequeño cambio en el servicio.
